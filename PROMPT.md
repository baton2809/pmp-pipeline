# PROMPT для GigaCode CLI — excel-parser (v3.1)

```
Прочитай SPEC.md и CONTRACT.md в корне репозитория целиком. excel-parser — первый из 4 скиллов pipeline pmp-vs-jira.

Сгенерируй два файла:

1. skill/excel-parser/SKILL.md  — главная инструкция для агента
2. skill/excel-parser/helper.py — вспомогательные Python-функции

Что делает скилл:
- Читает Бэклог и цели.xlsx, лист Q2_26_оценки_new_name (имена жёстко зафиксированы)
- Извлекает задачи через поиск колонок ПО ИМЕНИ ЗАГОЛОВКА (case-insensitive подстрока)
- Создаёт .cache/enriched.json по структуре из CONTRACT.md секция "После excel-parser"
- НЕ ходит в Jira

Критически важно — ЧИТАТЬ ВНИМАТЕЛЬНО:

1. ИСПОЛЬЗОВАТЬ ТОЛЬКО openpyxl. НЕ парсить xlsx как XML вручную. В предыдущей попытке (v3) был ручной XML-парсинг через `findall('{http://schemas.openxmlformats.org/spreadsheetml/2006/main}c')` — это привело к сдвигу колонок и из 28 задач нашлось 3. См. раздел 11 SPEC.md "Антипаттерны/Критические".

2. ИСКАТЬ КОЛОНКИ ПО ИМЕНИ ЗАГОЛОВКА, не по фиксированной букве (B, C, F, G...). См. раздел 5 SPEC.md. В предыдущей v3 я зафиксировал буквы и это сломалось. v2 (которая работала и нашла 28 задач) использовала поиск по имени — повторяем эту проверенную логику.

3. РАЗДЕЛ 7 SPEC.md УЖЕ СОДЕРЖИТ ГОТОВЫЙ КОД helper.py. Использовать его как основу. Все ключевые функции (open_workbook, find_header_row, detect_columns, parse_cr_key, extract_plan, clean_text, save_enriched, now_iso) — уже прописаны с правильной реализацией. Не изобретать заново.

4. Раздел 8 SPEC.md (формат SKILL.md): SKILL.md — это инструкция для агента в чате, не Python-программа. Шаги на естественном языке. helper.py — это набор функций которые SKILL.md просит вызвать.

5. Раздел 11 SPEC.md (антипаттерны): ТОЛЬКО один Python-файл с именем helper.py. Никаких main.py, process.py, run_*.py, generate_*.py, __pycache__, requirements.txt, виртуальных окружений. В прошлой v3 GigaCode создал 4 файла и сломал pipeline — не повторять.

6. parse_cr_key использует re.SEARCH, не re.match и не re.fullmatch. Потому что ключ может быть в URL ('https://jira.delta.sbrf.ru/browse/CRSIGMA-26516') а не в начале строки. См. раздел 5 SPEC.md.

7. Колонки 'аналитика', 'разработка', 'тестирование' встречаются в файле ДВАЖДЫ — это две версии плана. extract_plan берёт ПОСЛЕДНЕЕ непустое значение для каждой фазы. См. раздел 7 SPEC.md (готовая реализация).

8. CONTRACT.md секция "После excel-parser" — структура json строго оттуда. Не выдумывать поля.

9. Никаких MCP-вызовов. Этот скилл с Jira не общается.

Когда сгенерируешь — выведи в чат:
- Структуру созданных файлов (должно быть РОВНО 2 файла: SKILL.md и helper.py)
- Список функций в helper.py (должны быть: open_workbook, get_sheet, find_header_row, detect_columns, parse_cr_key, clean_text, extract_plan, save_enriched, now_iso)
- Один пример Step из SKILL.md (чтобы было видно что это инструкция, не Python-код)
- Подтверди что используется только openpyxl (не ручной XML)
- Подтверди что поиск колонок идёт по имени заголовка, не по фиксированной букве
- Подтверди что используется re.search (не match/fullmatch)
- Проверь что нет лишних .py файлов
```

## Что изменилось в v3.1 относительно v3

| Что | v3 (не сработало) | v3.1 (новая) |
|-----|-------------------|--------------|
| Поиск колонок | По фиксированной букве (B, C, F, G...) | По имени заголовка (case-insensitive подстрока) |
| Парсинг Excel | Допускался ручной XML-парсинг | **Только** openpyxl |
| Готовый код helper.py | Только описание функций | **Готовый код** в разделе 7 SPEC.md |
| Парсинг CR | Не было указания на re.search | Явно указано использовать re.search |
| Версии плана | "Если R/S/T все три заполнены" | Берётся последнее непустое значение для каждой фазы независимо |
