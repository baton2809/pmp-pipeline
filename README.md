# pmp-pipeline — 4 скилла для контроля портфеля задач

ETL-pipeline из 4 независимых скиллов для GigaCode CLI. Каждый скилл делает одну вещь, скиллы передают данные через `.cache/enriched.json` в рабочей директории.

**Главная задача:** дать архитектору (Наталье) автоматический контроль соответствия плана задач квартала (ПМП-Excel) с реальным состоянием в Jira — статус, эпик, команда, **факт А/Р/Т в календарных днях**, маркеры "застрявших".

## Pipeline

```
┌─────────────────────────────────────────────────────────────────────┐
│ 1. excel-parser                                                     │
│    Читает: Бэклог и цели.xlsx (жёстко зафиксирован)                 │
│    Создаёт: .cache/enriched.json (план задач)                       │
│    Без MCP                                                          │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 2. jira-enricher                                                    │
│    Читает: .cache/enriched.json                                     │
│    MCP: jira_get_issue (БЕЗ changelog) × N + jira_search × M эпиков │
│    Дополняет: статус, фаза, эпик, команда, lead_time                │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 3. timing-analyzer (опционально)                                    │
│    Читает: .cache/enriched.json                                     │
│    MCP: jira_get_issue С expand=changelog для active задач          │
│    Дополняет: phase_days {A, R, T} в календарных днях               │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 4. report-builder                                                   │
│    Читает: .cache/enriched.json                                     │
│    Создаёт: report.md                                               │
│    Без MCP                                                          │
└─────────────────────────────────────────────────────────────────────┘
```

## Архитектурные решения

**Почему 4 скилла а не один:**
- Контекст агента не переполняется (каждый скилл работает с маленькой задачей)
- Каждый скилл легко перезапустить отдельно (обновить только jira-статусы — перезапуск jira-enricher, остальное не трогаем)
- Можно итерировать на одном скилле не ломая остальные
- Расширение для v4/v5/v6 — добавляем новые скиллы в pipeline без изменения существующих

**Почему `enriched.json` один на всех:**
- Проще для дебага (один файл — открыл и видишь всё)
- Не нужна синхронизация между несколькими файлами
- Можно остановиться после любого скилла и посмотреть промежуточный результат

**Почему запрет на лишние .py файлы:**
- В прошлой v3 GigaCode создал 4 файла (`main.py`, `process.py`, `generate_report.py`, `run_jira_v3.py`) и pipeline сломался
- Один `helper.py` на скилл — это функции которые SKILL.md просит вызвать. SKILL.md остаётся главной точкой входа

## Подготовка окружения

### MCP

В `~/.gigacode/settings.json` для Atlassian:

```json
"includeTools": ["jira_get_issue", "jira_search"]
```

### Файлы в рабочей директории

```
<рабочая_директория>/
├── Бэклог и цели.xlsx       ← файл Натальи (жёстко зафиксированное имя)
└── .cache/                  ← создаётся автоматически при первом запуске
    └── enriched.json
```

## Установка скиллов

Для каждого из 4 скиллов: one-shot генерация → проверка → копирование в gigacode.

### Скрипт установки всех (после one-shot всех 4)

```bash
# В корне pmp-pipeline:
for skill in excel-parser jira-enricher timing-analyzer report-builder; do
    cp -r $skill/skill/$skill ~/.gigacode/skills/
done
```

## Запуск pipeline

В рабочей директории (где лежит `Бэклог и цели.xlsx`):

```bash
gigacode
```

В чате последовательно:

1. `запусти excel-parser` → создан `.cache/enriched.json` с планом
2. `запусти jira-enricher` → дополнен статусами / эпиками / командами
3. `запусти timing-analyzer` → дополнен фактом А/Р/Т (опционально)
4. `запусти report-builder` → создан `report.md`

Каждый шаг проверяет что предыдущий отработал. Если запустить из середины — сообщит что нужно сначала.

**Время полного pipeline:** 2-4 минуты (28 задач × ~3-4 вызова MCP).

## Что показать Наталье

После прогона открыть `report.md`. Главные разделы для разговора:

| Раздел | О чём говорить |
|--------|----------------|
| "Что в отчёте и чего нет" | Проговорить ограничения: календарные дни ≠ чел-дни, маркер ⚠ = сигнал, не точный факт |
| "Срез по эпикам" | Обзор: сколько задач в плане под каждой инициативой, сколько всего в эпиках |
| "Срез по командам" | Кто чем загружен |
| "Застрявшие" (если есть timing) | **Главная польза** — конкретный список задач для разбора с командой |

Спросить у Натальи:
- Это полезный взгляд?
- Какие пороги маркера ⚠ лучше (сейчас × 2 от плана)?
- Включать ли v4 (контроль плановых дат ИФТ/ПСИ/ПРОМ)?

## Идемпотентность и обновления

Любой скилл можно запускать повторно. `enriched.json` перезаписывается частично — только в полях которые обрабатывает данный скилл.

**Сценарии:**
- Обновился `Бэклог и цели.xlsx` → весь pipeline сначала
- Обновились статусы в Jira → `jira-enricher` + `timing-analyzer` + `report-builder`
- Только обновить отчёт → только `report-builder`

## Удаление кеша

```bash
rm -rf .cache/
```

После этого pipeline нужно запускать с `excel-parser`.

## Принципы дизайна (важно)

1. **Один скилл — одна вещь.** Если хочется добавить функцию — лучше новый скилл, не раздувать существующий.
2. **CONTRACT.md — единый источник истины** для структуры `enriched.json`. Изменения в структуре — сначала в CONTRACT.md, потом в коде.
3. **Только helper.py.** Запрещены `main.py`, `process.py`, `run_*.py`, `generate_*.py`, `__pycache__`, виртуальные окружения. SKILL.md остаётся главной точкой входа.
4. **MCP-вызовы — нативные tool calls агента**, не Python-обёртки. Никаких `result = mcp_jira_get_issue(...)`.
5. **Жёстко зафиксированы:** имя Excel-файла, имя листа, колонки B/C/F/G/J/K/L/R/S/T. Не искать по именам заголовков (нестабильно).
6. **Календарные дни ≠ чел-дни** — всегда дисклеймер. Не выдумывать точный факт где его нет.

## Что делать если что-то сломалось

| Проблема | Что делать |
|----------|------------|
| Скилл создал `main.py` или другие лишние .py | Удалить, перегенерировать с явной ссылкой на запрет в SPEC.md |
| Скилл сделал заглушку `# здесь будет вызов MCP` | Перегенерировать, явно указать на раздел "КРИТИЧНО формат MCP" в SPEC.md |
| Контекст переполнился | Пайплайн как раз решает эту проблему — отдельные скиллы. Если в одном скилле переполнение — урезать `fields=`. |
| `.cache/enriched.json` пустой | Проверить что excel-parser отработал, проверить логи |
| MCP-вызов не работает | Проверить `/mcp` в gigacode, проверить токен, проверить `includeTools` |
| Маркеры выглядят странно | Это сигнал что пороги не подходят. Обсудить с Натальей, скорректировать SPEC report-builder. |
