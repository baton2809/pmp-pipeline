# report-builder v3.1 — четвёртый (последний) скилл pipeline

Генерирует финальный markdown-отчёт `report.md` из накопленного `pipeline/enriched.json`. Не ходит в Jira, не модифицирует данные — только формирует текст.

## Изменения v3.1 vs v3.0

- ✅ Читает из `pipeline/` вместо `.cache/`
- ✅ Готовый код `helper.py` в SPEC.md раздел 5 — целиком работающий
- ✅ `report.md` создаётся в **корне** рабочей директории (не в `pipeline/`), как финальный артефакт
- ✅ Минорно обновляет `enriched.json` (только `report_generated_at` и `skills_completed`)

## Место в pipeline

```
1. excel-parser
2. jira-enricher
3. timing-analyzer (опционально)
4. report-builder    ← вы здесь    → report.md в корне
```

## Условный рендеринг

Скилл работает в двух режимах:

**Полный режим** (timing-analyzer отработал):
- Все 7 секций
- Колонки "Факт А/Р/Т"
- Секция "Застрявшие"

**Без timing режим** (timing-analyzer не запускался):
- 6 секций (без "Застрявшие")
- Без колонок "Факт А/Р/Т"
- В дисклеймере пометка "для расчёта факта запустите timing-analyzer"

## Подготовка

MCP не нужен.

Должны отработать как минимум `excel-parser` и `jira-enricher`. `timing-analyzer` опционален.

## Запуск

### Шаг 1. Сгенерировать

```bash
cd report-builder
gigacode
```

Вставить промпт из `PROMPT.md`.

### Шаг 2. Проверить перед установкой

В `skill/report-builder/` — ровно 2 файла:

- helper.py содержит функции: `truncate_words`, `compute_marker`, `compute_summary`, `build_report`
- SKILL.md не делает MCP-вызовов
- SKILL.md не модифицирует `pipeline/enriched.json` сильнее чем `report_generated_at` + `skills_completed`
- Условный рендеринг (`if timing_available`) реализован
- `truncate_words` обрезает по словам через `rsplit(' ', 1)[0]`, не посередине слова
- UTF-8 без BOM (используется `ensure_ascii=False`)

### Шаг 3. Установить и запустить

```bash
cp -r skill/report-builder ~/.gigacode/skills/
gigacode
```

В чате: "запусти report-builder".

После прогона:
- `report.md` создан в корне рабочей директории
- В чате сводка с указанием режима и размера

## Что показать Наталье

Открыть `report.md` в редакторе с поддержкой markdown (VS Code, Obsidian, Telegram, GitHub).

Главные секции для разговора:

1. **Дисклеймер** ("Что в отчёте и чего нет") — проговорить ограничения сразу
2. **Секция 5 "Застрявшие"** (если есть timing) — конкретный список для разбора с командой
3. **Секция 3 "Срез по эпикам"** — обзор где сколько работы
4. **Секция 4 "Срез по командам"** — кто чем загружен

Вопросы для Натальи:
- Это полезный взгляд на портфель?
- Что добавить/убрать из таблицы?
- Какие пороги маркера ⚠ лучше (сейчас × 2 от плана)?
- Хочется ли v4 (контроль дат ИФТ/ПСИ/ПРОМ)?

## Сколько занимает

~2-5 секунд. Никаких внешних вызовов, только локальная обработка json.

## Повторный запуск

Можно запускать сколько угодно раз — отчёт перезаписывается. Если в `enriched.json` ничего не изменилось — отчёт будет идентичным.

Если хочется обновить отчёт с новыми данными — сначала перезапустить `jira-enricher` (и опционально `timing-analyzer`), потом `report-builder`.

## Если что-то пошло не так

| Симптом | Действие |
|---------|----------|
| Не строится отчёт без timing | Условный рендеринг не реализован — проверить `if timing_available` в helper.py |
| Имена обрезаются посередине слова | `truncate_words` не использует rsplit — перегенерировать |
| Кириллица как `\u...` | Нет `ensure_ascii=False` — перегенерировать |
| Маркер ⚠ ставится странно | Проверить пороги в `compute_marker` |
| report.md в `pipeline/` вместо корня | Жёстко указать путь `'report.md'` в build_report, не `'pipeline/report.md'` |
